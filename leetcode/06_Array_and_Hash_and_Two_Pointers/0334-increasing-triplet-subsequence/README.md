# 334. Increasing Triplet Subsequence

## 我的思考
初始思路是维护两个下标：
- `one` 记录当前递增子序列的第一个数下标
- `two` 记录第二个数下标

遍历时：
- 如果找到比 `two` 更大的数，直接返回 `true`
- 如果当前数小于等于 `one`，则更新 `one`
- 如果当前数大于 `one` 且小于等于 `two`，则更新 `two`

也就是始终维护：
- 一个尽可能小的长度为 2 的递增候选序列（为了找到答案）
- 一个尽可能小的长度为 1 的递增候选序列（为了更新长度为2的候选序列）

初版先用 `while` 预处理，找到第一个满足递增关系的 `oneIndex` 和 `twoIndex`，之后再继续扫描。思路本身没有问题，时间复杂度 `O(n)`、空间复杂度 `O(1)`，但可读性一般，而且这一步预处理其实不是必须的。

优化后，直接把下标改成数值维护：
- `oneNum` 表示当前最小的第一个数
- `twoNum` 表示当前最小的、且大于 `oneNum` 的第二个数

遍历每个 `num` 时：
- 若 `num > twoNum`，说明已经找到第三个数，返回 `true`
- 若 `num > oneNum`，则可以尝试把 `twoNum` 更新为更小的第二个数
- 否则更新 `oneNum`

这样写更简洁，也更容易看出本质。

一开始没有直接想到这一版，主要卡在 `two` 的初始化。  
如果直接把 `two` 初始化成 `nums[1]`，那么像 `[1,0,1]` 这样的情况就会出错，因为此时 `two` 还没有真正成为一个合法的“第二个数”。

后面想到，直接把 `twoNum` 初始化为 `Integer.MAX_VALUE` 即可。  
这样只有在真正遇到一个大于 `oneNum` 的数时，`twoNum` 才会被赋成合法候选值，也就不会出现 “`two` 还没合法，就被拿去比较” 的问题。

## 卡点
1. 初版思路能做出来，但用了额外的 `while` 预处理，结构不够简洁。
2. 一开始没有直接想到用两个数值而不是两个下标来维护状态。

## 关键点
1. 始终维护尽可能小的 `oneNum` 和 `twoNum`。
2. `oneNum < twoNum` 表示当前已经有一个长度为 2 的递增候选。
3. 一旦出现 `num > twoNum`，就说明存在递增三元子序列。
4. `twoNum` 初始化为 `Integer.MAX_VALUE`，可以避免非法初始状态带来的错误。

## 复杂度
- 时间：`O(n)`
- 空间：`O(1)`

## 注意
1. 优化版本质上是在维护长度为 1 和长度为 2 的最优递增候选。
2. `twoNum` 不能随便初始化为某个数组元素，否则在它尚未真正合法时就可能误判。
3. 这一题基本没有进一步的复杂度优化空间，主要优化点在可读性和状态表达。
