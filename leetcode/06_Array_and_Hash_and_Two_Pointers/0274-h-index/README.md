# 274. H-Index

## 我的思考
初始思路有两个。

思路 1 是直接排序，然后二分查找。  
如果 `nums[i] > 左边长度`，就往右边找，否则往左边找。这个做法时间复杂度是 `O(nlogn)`，可以做，但不像最优。

思路 2 是直接遍历，设 `HIndex[i]` 表示“有多少篇论文引用数大于等于 `i`”。  
主要问题在于更新这一步：如果对于引用指数 `x`，去更新所有小于等于 `x` 的 `HIndex[]`，时间复杂度会直接爆炸。

因此改成只更新“能更新的最大的那个位置”。  
先统计每个引用数出现的频次，然后从大到小遍历 `HIndex[i]`，并在遍历过程中做前缀累加：

- 若当前 `HIndex[i] >= i`，则直接返回 `i`
- 否则执行 `HIndex[i - 1] += HIndex[i]`

这样就能逐步得到“引用数大于等于某值的论文篇数”。

初版实现中，数组大小开到了 `maxCitation + 1`，因此时间复杂度是 `O(n + m)`，其中 `n` 是 `citations` 长度，`m` 是 `citations` 最大值。

之后 AI 提醒：空间复杂度可以继续压缩。

优化后才意识到，**最大的关键点不是 `maxCitation`，而是答案域**。  
H-index 的答案一定落在 `[0, n]`，而不是 `[0, maxCitation]`。因为论文总数只有 `n` 篇，不可能存在大于 `n` 的 H-index。

所以可以直接把所有大于 `n` 的引用数都压到 `n` 上，也就是：

`freq[Math.min(citation, len)]++`

这样频次数组只需要开到 `len + 1`，时间复杂度变成 `O(n)`，空间复杂度也压到了 `O(n)`。

这题最大的收获是：  
**要先搞清楚真正有价值的答案域，而不是只看到一个合法的取值范围。**  
前者是在考虑全局约束，后者只是满足最低约束。

## 卡点
1. 第一时间没有想到答案域其实是 `[0, n]`。
2. 初版自然地把频次数组开到了 `maxCitation + 1`，虽然能做，但不够优。
3. 没有立刻想到 `freq[Math.min(citation, len)]++` 这个压缩方式。

## 关键点
1. `HIndex[i]` / `freq[i]` 的含义是：引用数为 `i`，或最终累计后表示引用数大于等于 `i` 的论文篇数。
2. 从大到小累计频次，可以快速得到“至少有多少篇论文引用数不小于当前值”。
3. H-index 的答案域是 `[0, n]`，因为论文总数只有 `n`。
4. 所有大于 `n` 的引用数，对答案的贡献都等价于 `n`。

## 复杂度
- 时间：`O(n)`
- 空间：`O(min(n, m))` m为最大引用数

## 注意
1. 这题最重要的不是计数本身，而是先识别出真正的答案域。
2. `freq[Math.min(citation, len)]++` 是优化核心。
3. 对于“答案不可能超过某个全局上界”的题，要优先思考能否按答案域建模，而不是按原始数据范围建模。
