# 139. Word Break

## 我的思考
一开始先想到的是 `Trie` 匹配。  
但很快意识到，仅仅从前缀匹配角度出发并不够。例如：

- `s = "aabb"`
- `wordDict = ["aab", "aa", "bb"]`

这里无法仅根据当前前缀，直接确定前半部分应该选 `aab` 还是 `aa`。  
也就是说，这题不是单纯的贪心匹配，而是需要记录“某个位置之前是否可达”，因此进一步转向了 **DP / 状态机** 的建模。

具体做法是：
- `dp[i]` 表示前 `i` 个字符是否可以被拆分
- `dp[0] = true`
- 枚举每一个可达位置 `i`
- 从该位置出发，在 `Trie` 上继续匹配后续字符
- 每当匹配到一个单词结尾，就将对应的 `dp[j + 1]` 置为 `true`

也就是把 `Trie` 当作“从某个起点出发，快速枚举所有可匹配单词”的工具，而真正负责状态判定的是 `dp`。

## 卡点
1. 写 `Trie.insert()` 时卡了一下，忘记了只有在 `curr.next[c]` 不存在时才需要新建节点。
2. 在 `for (int j = i; j < len; j++)` 的匹配过程中又卡了一次。对 `Trie` 里“边存储什么、节点存储什么”的把握还不够自然，不能立刻形成清晰图景，仍然需要在脑中额外整理一遍。

## 关键点
1. 这题本质不是前缀匹配，而是**可达性 DP**。
2. `dp[i]` 表示前 `i` 个字符能否拆分成功。
3. `Trie` 的作用不是替代 DP，而是加速“从某个起点出发，能匹配到哪些单词结尾”。
4. 当 `dp[i] == true` 时，才有必要从 `i` 出发继续在 `Trie` 中尝试匹配。
5. 只要当前匹配位置是某个单词结尾，就可以把对应位置标记为可达。

## 复杂度
- 时间：`O(n^2)`，最坏情况下每个可达位置都要继续向后匹配
- 空间：`O(n + 总字符数)`，其中 `n` 为 `dp` 数组开销，`Trie` 占用字典总字符数级别空间

## 注意
1. `Trie` 节点存储的是“走到当前位置后的状态”和“是否为单词结尾”；边本质上对应字符转移。
2. 和上次“硬匹配”相比，这次的写法更接近“DP + Trie”结合的标准思路。
