# 72. Edit Distance

## 我的思考
这题直接识别为 DP。

状态转移也比较直接，因此在想到更新方式之后，滚动数组优化也能顺势看出来。

实现上使用了一维 `dp` 数组，其中 `dp[i + 1]` 表示当前处理状态下的编辑距离。  
在滚动更新时，除了当前行和前一行的常规覆盖关系，还需要额外保留左上角的值，也就是二维写法中的 `dp[i - 1][j - 1]`。这一点是滚动数组里最需要注意的地方。

代码里最后用了 `temp` 来暂存这个值。  
本质上，它表示的就是“左上角旧值”，只是当时没有第一时间想到一个更贴切的变量名。

另外一个点是，`dp` 数组最后选择了**多开一位**。  
这样可以自然表示空串状态，初始化也会清晰很多。  
如果不拓展这一位，边界和预处理都会变得比较麻烦。

## 卡点
1. 滚动数组需要额外记录左上角的旧值，即二维状态里的 `dp[i - 1][j - 1]`，当时没有第一时间想到合适的变量名，最后用了 `temp`。
2. 一开始没有立刻想到数组要多开一位；如果不这样做，预处理会比较麻烦。

## 关键点
1. 这题是经典编辑距离 DP。
2. 状态转移可由插入、删除、替换三种操作直接得到。
3. 滚动数组优化时，除了当前格依赖的左、上，还要额外保存左上角旧值。
4. `dp` 多开一位后，空串边界会自然很多。

## 复杂度
- 时间：`O(mn)`
- 空间：`O(m)`

## 注意
1. `temp` 这个变量语义其实更接近“leftUp / diagonal / preDiag”，以后固定为preDiag。
2. 这类字符串 DP，状态数组通常都应该优先考虑多开一位。

## 补充思考

这题对应到此前 DP 专题总结的一套通用分析流程：

1. 从题目中抽离出 DP 的三要素：**初始值、更新方式、答案位置**。  
   一旦这三点明确，就可以基本抛开题目背景，专注于 DP 数组本身的推导。

   例如本题：
   - 初始值  
     - `dp[0][i] = i`  
     - `dp[j][0] = j`
   - 更新方式  
     - `dp[i][j] = min(`  
       `dp[i - 1][j] + 1,`  
       `dp[i][j - 1] + 1,`  
       `dp[i - 1][j - 1] + (isMatch ? 0 : 1)`  
       `)`
   - 答案位置  
     - `dp[m][n]`（右下角）

2. DP 时优先考虑是否需要**拓展一位**。  
   当状态需要描述“空状态”时，多开一行或一列通常能显著简化初始化与转移逻辑。本题中通过多开一位，自然表示空字符串的编辑距离。

3. 根据更新方式判断**遍历方向与是否可以压缩空间**。  
   本题中：
   - `dp[i][j]` 依赖  
     - `dp[i - 1][j]`（上）  
     - `dp[i][j - 1]`（左）  
     - `dp[i - 1][j - 1]`（左上）
   - 因此遍历方向必须从左到右，保证 `dp[i][j - 1]` 已经更新完成
   - 同时只依赖当前行和上一行，可以进行滚动数组压缩
   - 压缩后仍需额外保存左上角值 `dp[i - 1][j - 1]`

整体流程为：
- 先抽象 DP 三要素  
- 判断是否需要扩展一位  
- 根据依赖关系确定遍历方向  
- 判断能否压缩并确定需要额外保存的状态
