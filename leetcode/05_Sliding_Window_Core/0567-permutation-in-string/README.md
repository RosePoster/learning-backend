# 567. Permutation in String

## 我的思考
一开始的思路是维护一个 `int[26]` 的 `count` 数组，统计字母频率，并先用 `s1` 初始化这个数组。

例如当 `s1 = "ab"` 时：
- `count[0] = -1`
- `count[1] = -1`

这表示当前还缺一个 `a` 和一个 `b`。

然后使用滑动窗口，在 `s2` 上维护 `left` 和 `right` 两个指针进行遍历：
- `right` 前进后，执行 `count[s2.charAt(right)]++`
- `left` 前进后，执行 `count[s2.charAt(left)]--`

其中：
- `left` 前进的条件是 `count[s2.charAt(left)] > 0`，也就是当前窗口中 `left` 指向的字母是多余的，不需要保留
- `right` 前进的条件是当前窗口还不是答案

对于“当前窗口是否是答案”的判断，最初的做法是遍历整个 `count` 数组，只要每一项都为 `0`，就说明当前窗口和 `s1` 完全匹配。

进一步地，触发检查的条件也有过思考。窗口变动后，只有在某个被修改的位置变成 `0` 时，才更有可能接近答案。例如假设 `right` 前进后，`count[s2.charAt(right)]++` 的结果变成 `0`，就触发一次检查。

但是每次检查都遍历整个 `count` 数组，那么单次是 `O(26)`。虽然常数不大，但仍然有额外开销。

因此引入一个 `unsatisfied` 变量，用来记录还有多少个字母条件没有满足。某个 `count` 值归零后，就执行 `unsatisfied--`；如果某个值从满足状态重新变为不满足，则对应执行 `unsatisfied++`。当这个变量归零时，说明窗口满足要求，返回 `true`；否则最终返回 `false`。

在实现完这一版之后，又进一步发现：**这个题的滑动窗口长度其实可以固定为 `s1.length()`**。

因为题目要求的是 `s2` 中是否存在一个子串，它恰好是 `s1` 的排列。排列的长度一定和 `s1` 相同，因此没有必要让窗口动态伸缩，只需要维护一个固定长度窗口即可。

于是又写了第二版。

第二版的做法是：
- 先用 `s1` 对 `count` 做减法
- 同时用 `s2` 的前 `s1.length()` 个字符对 `count` 做加法

这样初始化完成后，`count` 表示“当前固定窗口”和 `s1` 在各字符上的差值。

接着再统计 `unsatisfied`，但此时它的语义已经变化了：
- 第一版里，`unsatisfied` 表示“还有多少个字母条件没有满足”
- 第二版里，`unsatisfied` 表示“当前还有多少个字母频次不匹配”

之后窗口每次右移一格：
- 左侧字符移出，更新一次 `count`
- 右侧字符移入，再更新一次 `count`

每次更新时，都根据该字符对应位置是否从 `0` 变成非 `0`，或者从非 `0` 变成 `0`，来调整 `unsatisfied`。只要 `unsatisfied == 0`，就说明当前固定窗口和 `s1` 完全匹配，返回 `true`。

两版代码都能通过 LeetCode，但第二版更符合这道题的本质，因为窗口长度天然固定，思路也更直接。

## 卡点
1. 初版思路默认把它当成一般的“可伸缩滑动窗口”问题处理，没有第一时间意识到排列子串的长度一定固定。
2. 最初判断窗口是否为答案时，需要每次遍历整个 `count` 数组，虽然是常数复杂度，但仍然显得不够干净。
3. 在引入 `unsatisfied` 之后，需要非常清楚它的语义，否则在更新 `count` 时容易写错增减逻辑。
4. 第二版虽然更优，但 `unsatisfied` 的语义已经从“还差多少”变成了“还有多少项不匹配”，这一点如果不明确，容易和第一版混淆。

## 关键点
1. 这道题的本质是**固定长度滑动窗口**，因为排列后的字符串长度不变。一般地，题目给了一个目标串/模式，要在另一个串里找它的某种变体（排列、异位词等）。看到这个结构，第一反应就应该是固定窗口。

## 复杂度
- 时间：`O(n)`
- 空间：`O(1)`

## 注意
1. 第二版比第一版更好，因为窗口长度固定，思路更贴题。
2. `unsatisfied` 的定义必须先想清楚，再写更新逻辑，否则很容易在 `0 / 非0` 的边界变化上出错。
3. 第一版虽然也能通过，但在这道题上属于“能做出来，但没有完全利用题目结构”的写法。
