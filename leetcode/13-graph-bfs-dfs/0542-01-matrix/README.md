# 542. 01 Matrix

## 我的思考
最开始想到的是 `dfs`。

理由是：一个节点的答案依赖其上下左右 4 个方向的最短距离，因此似乎可以通过深度遍历，递归求出当前节点的最短距离。为此先写了一个记忆化搜索：
- `ans[i][j] == -1` 表示还未求出
- 如果当前位置是 `0`，则答案为 `0`
- 如果当前位置是 `1`，则答案为四个方向最小值加 `1`

实现过程中打了一个补丁：

`ans[i][j] = Integer.MAX_VALUE`

这样做是为了避免递归过程中回头再次访问自己。

但这个补丁并不能真正修复问题。  
原因在于：某个节点依赖的其它节点，虽然理论上应当共享某条最短路径，但这些节点本身的最短距离可能还没有被正确求出来。这样当前节点就会基于一个未完成、未收敛的状态提前得到结果，之后也无法再更新成真正最短距离。

因此，这个 `dfs` 思路本质上不成立。耗时 20 分钟后发现答案错误，又思考了 3 分钟，判断这个方向无法自然修正，于是换思路。

第二个思路是 `bfs`，即从所有 `0` 往外一层层推进。

一开始没有直接使用 `bfs`，是因为担心某个节点不能直接更新周围节点。直觉上觉得它有点像“路由距离”问题，似乎每个点都需要先比较四周所有节点的最短距离，才能确定自己的值，因此最开始更偏向 `dfs`。

但重新审视后发现，这道题其实正适合 `bfs`。  
原因在于队列是有层次顺序的：

- 距离为 `1` 的节点一定先入队、先处理
- 距离为 `2` 的节点只能由距离为 `1` 的节点推出，因此一定排在后面
- 更远的节点同理

所以某个节点第一次被更小距离更新时，这个距离就已经是当前最优层次传播下的结果。  
也就是说，`bfs` 天然保证了“按距离从小到大扩展”，不需要像 `dfs` 那样等待四周状态全部收敛后再回头修正。

最终写出的做法是：
- 先把所有 `0` 位置置为 `0`
- 把所有与 `0` 相邻的 `1` 作为初始层入队，置为 `1`
- 然后不断从队列中取点，尝试用当前距离 `+1` 更新四邻
- 如果某个邻居能被更新成更小值，就入队继续扩展

这一版耗时 14 分钟，AC。

## 卡点
1. 一开始把这题理解成“当前点依赖四周最优值”的递归问题，因此优先选择了 `dfs`。
2. 在 `dfs` 中虽然尝试用 `Integer.MAX_VALUE` 防止回头递归，但这只能避免死循环，不能解决“依赖节点答案尚未收敛”的根本问题。
3. 对 `bfs` 的顾虑在于，最初没有立刻意识到队列顺序天然保证了按最短距离逐层扩展。

## 关键点
1. 这题本质是 **multi-source BFS**，不是 `dfs`。
2. 所有 `0` 都可以看作距离源点，最短距离问题等价于“从所有 0 同时出发向外扩散”。
3. `bfs` 的队列顺序保证了：距离小的层一定先处理，因此后续节点第一次被正确更新时，就是按层推进得到的最短距离。
4. `dfs` 不适合这题，因为节点之间存在相互依赖，递归过程中很容易在依赖值尚未稳定时提前写入错误答案。

## 复杂度
- 时间：`O(mn)`
- 空间：`O(mn)`

## 注意
1. 更标准的写法通常是：**直接把所有 `0` 一起入队**，然后统一做 multi-source BFS。这样比“先找与 0 相邻的 1 作为第一层”更自然。
2. 这题里 `bfs` 能成立的根本原因，不是“某个点能不能更新周围”，而是“队列保证了更新顺序是按距离递增的”。
3. `dfs + memo` 常适合无环依赖或可明确拓扑化的问题；这题这种最短路型扩散问题，更适合 `bfs`。
