# 318. Maximum Product of Word Lengths

## 我的思考
这题第一眼的思路是：

用 `words` 建立 Trie（前缀树），遍历 `words`，对每一个 `word` 搜索 Trie，用 `set` 记录当前 `word` 里的字母。在搜索 Trie 的过程中跳过 `set` 中记录字母对应的分叉，从而实现“无公共字母”的约束。

AI 提示：“这个第一眼思路，方向不优。可以用掩码。”

于是开始反思 Trie 是否可行。

掩码方法其实也考虑过，但当时的问题在于：两两检查是否相交是 `O(n^2)`。

而 Trie 的方法看起来像是：搜索一棵树的最大高度，约束是 `set` 里的路径不能走。  
于是每一个 `word` 的答案就是 `word.length * h`，其中 `h` 是约束下搜索到的最大高度。  
表面上看，时间复杂度似乎可以写成 `O(n * h)`，其中 `n` 是 `words` 的大小，`h` 是单词最大长度，看起来似乎优于平方复杂度。

但继续往下想，会发现并不是这样。  
因为搜索时并不只是走一条路径，而是需要遍历禁用分叉之外的所有分叉。  
这里的 `h` 在最坏情况下并不只是单条路径长度，而更接近 `k * h`，其中 `k` 是所有 `word` 的规模总和。虽然 `set` 会禁用一部分路径，不至于完全达到这么夸张的程度，但仍然会显著大于单纯的 `h`。

于是继续想：有没有剪枝？  
例如，在每一个节点记录本节点最大深度，对于已经没有潜力的节点直接跳过。  
这样时间复杂度是否有可能优于 `n^2`？

AI 回答：“但仍然不如朴素 `n^2`。”

于是决定先实现代码，先写出了 `int[] bitMask = buildMask(words);` 再枚举的版本。

接着重新思考时间复杂度：  
最坏情况下，如果所有 `word` 两两之间都没有公共字母，那么每一对都必须检查，无法跳过任何一对。  
也就是说，最坏情况下的两两比较本身就是不可避免的。

如果要尝试跳过重复，本质上就是想走 Trie 那种“把重复字符编在一起”的思路。

这时 AI 提示：“不，字符范围是小写字母，只有 26 个，这些小写字母根本撑不起多大的最坏空间。”

这一下就把关键点点出来了。  
按 `mask` 聚类，将 `n` 缩小为 `m`，是非常有潜力的优化。

于是先确认哪些情况可以合并：

`10110` 可以吞并 `00110` 吗？不可以。  
必须是**精确相同**的 `mask` 才能合并。  
因此可以使用一个 `Map`，键为 `mask`，值为该 `mask` 下的最大长度。

AI 还提示了一个更激进的优化：“`mask` 之间存在超集关系，比如 `10110` 和 `00110`，如果 `10110` 对应的 `word` 更长，那 `00110` 作为候选永远不会赢。因为任何能和 `10110` 配对的 `word`，也能和 `00110` 配对，但长度更短。”

继续分析后，发现这个结论不对。

假设 `10110` 和 `00110`，如果 `10110` 对应的 `word` 更长，那么：
- 能和 `10110` 配对的 `word1`，确实也能和 `00110` 配对；
- 但能和 `00110` 配对的 `word2`，**不一定**能和 `10110` 配对；
- 而且 `word2` 和 `00110` 配对得到的结果，完全有可能大于 `10110` 和 `word1` 配对得到的结果。

所以，只有**精确相同的 `mask`** 才能安全合并。  
这一点已经能合并很多情况了，比如 `a`、`aa`、`aaa`、`aaaa` 都会合并成同一个 `mask`。

实现时还有一个卡点：`Map` 如何转成 `List`，以及 `entrySet` 等语法不太清晰。

## 卡点
1. 一开始误以为 Trie 搜索在加上字母禁用约束后，可能优于两两枚举。
2. 没有立刻意识到：最坏情况下所有合法对都必须检查，`O(n^2)` 的比较下界很难突破。
3. 一度考虑过基于 `mask` 超集关系继续剪枝，但进一步分析后发现这种剪枝并不安全。
4. 实现层面卡在 `Map -> List` 的转换，以及 `entrySet` 的使用语法上。

## 关键点
1. 每个单词都可以压缩成一个 26 位二进制掩码（bitmask），表示出现过哪些字母。
2. 两个单词是否有公共字母，可以通过  
   `mask1 & mask2 == 0`  
   在 `O(1)` 时间内判断。
3. 只需要保留每个 `mask` 对应的**最大单词长度**，因为相同 `mask` 下更短的单词不可能更优。
4. 安全的合并只能发生在**精确相同的 `mask`** 之间，不能因为超集关系就直接删掉子集 `mask`。
5. 优化后的核心不是消灭两两比较，而是通过 `mask` 聚类，把原本的 `n` 个单词压缩成 `m` 个不同的 `mask`，从而将枚举规模从 `n^2` 降到 `m^2`。

## 复杂度
- 时间：`O(L + m^2)`，其中 `L` 是所有单词字符总数，`m` 是不同 `mask` 的数量；最坏情况下可视为 `O(L + n^2)`
- 空间：`O(m)`

## 注意
1. `buildMask` 的返回类型最好不要写原始类型 `List`，应写成  
   `List<Map.Entry<Integer, Integer>>`，否则会丢失泛型信息。
2. `currMask |= 1 << word.charAt(j) - 'a';`  
   这里依赖运算符优先级，实际含义是  
   `currMask |= 1 << (word.charAt(j) - 'a');`  
   加括号可读性更好。
3. `bitMasks.merge(currMask, word.length(), Math::max);` 的含义是：若 `currMask` 已存在，则保留更大的长度。
4. `new ArrayList<>(bitMasks.entrySet())` 得到的是所有 `Map.Entry<mask, maxLen>` 的列表，之后就可以通过  
   `getKey()` 取 `mask`，`getValue()` 取该 `mask` 的最大长度。
5. 这道题的核心不是“找两个具体单词”，而是“找两个字母集合不相交的状态”，因此用字符集合的状态压缩比字符串结构本身更合适。
